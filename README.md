# HPE Aruba Central Next Gen Collector for Zabbix

Python collector for HPE Aruba Networking Central Next Gen. It reads data from GreenLake/New Central APIs and pushes values to Zabbix trapper items with `zabbix_sender`.

The collector runs outside Zabbix, so it also works with Zabbix Cloud where local external scripts are not available.

## How It Works

1. The collector reads one or more GreenLake workspaces from `workspaces.json`.
2. For each workspace it creates an OAuth access token from `workspace_id`, `client_id`, and `client_secret`.
3. For MSP workspaces, it discovers MSP tenants and exchanges the MSP token for tenant tokens.
4. It polls New Central APIs for APs, radios, switches, switch interfaces, switch hardware/stack/LAG/VSX data, and GreenLake subscriptions.
5. It can plan or create Zabbix host groups and hosts through the Zabbix API.
6. It sends discovery and raw values with `zabbix_sender`.

The Zabbix template uses dependent items to extract metrics from raw master items. Raw items have `history: 0`; they feed dependent items but do not keep JSON history.

## GreenLake Credentials

For each workspace, collect:

- `workspace_id`
- `client_id`
- `client_secret`
- `central_base_url`

In GreenLake, open the target workspace and create or regenerate API client credentials. Store the client secret securely and never commit it to Git.

For MSP mode, use the MSP workspace ID. The collector will discover tenant workspaces through API calls. For standalone mode, use the customer workspace ID.

Set `central_base_url` to the New Central API cluster for the workspace, for example:

```text
https://de2.api.central.arubanetworks.com
```

Common clusters include `de1`, `de2`, `de3`, `gb1`, `us1`, `us2`, `us4`, `us5`, `us6`, `ca1`, `in1`, `jp1`, `au1`, and `ae1`. If the cluster is wrong, GreenLake authentication can still work while Central API calls return `401`, `400`, or empty data.

Official references:

- [HPE GreenLake API Client Credentials](https://developer.greenlake.hpe.com/docs/greenlake/services/credentials/public)
- [HPE GreenLake API Authentication](https://developer.greenlake.hpe.com/docs/greenlake/guides/public/authentication/authentication/)
- [HPE GreenLake Workspace Management](https://developer.greenlake.hpe.com/docs/greenlake/services/workspace/public)
- [HPE GreenLake Subscription Management](https://developer.greenlake.hpe.com/docs/greenlake/services/subscription-management/public/)
- [New Central Switch Monitoring APIs](https://developer.arubanetworks.com/new-central/reference/getswitch)
- [New Central Streaming APIs](https://developer.arubanetworks.com/new-central/docs/streaming-api-getting-started)

## Zabbix Setup

Import:

```text
zabbix_template_hpe_aruba_central_new_ap_trapper.yaml
```

Major release 1.x is moving from a single monolithic Zabbix host to a collector/site/device model:

- one collector host for collector health and package/config status;
- one site host for each mapped Central site;
- one device host for each AP, switch, or gateway.

The collector can create and update host groups and hosts through the Zabbix API. Device metric triggers are generated by Zabbix templates from collected items.

Template macros:

```text
{$CENTRAL.COLLECTOR.NODATA}                 default 15m
{$CENTRAL.LICENSE.EXPIRY.WARNING.DAYS}      default 30
{$CENTRAL.TEMPLATE.VERSION}                 package version
```

Central alerts are not used as the default alert source in 1.x. Standard Zabbix triggers generated from collected items are the primary alerting path. Central alert items may return later as optional informational items.

Discovered items include these tags:

- `tenant = {#TENANT_NAME}`
- `workspace = {#WORKSPACE_NAME}`
- `site = {#SITE_NAME}` where available
- `device_type = {#DEVICE_TYPE_TAG}`
- license items also include `license_scope`, `owner`, and `license_type`

`{#DEVICE_TYPE_TAG}` is generated by the collector, because Zabbix does not expand user macros in LLD item prototype tag values.

## Collector Setup

Copy the project to a server or PC that can reach:

- GreenLake/New Central APIs over HTTPS
- Zabbix server or proxy on trapper port `10051`

Requirements:

- Zabbix 7.0 or newer.
- Python 3.10 or newer.
- `zabbix_sender`.

On Windows, `zabbix_sender.exe` can stay in the collector folder.

Copy the example config:

```powershell
Copy-Item .\workspaces.example.json .\workspaces.json
```

Example:

```json
{
  "config_version": "1.0.1",
  "zabbix": {
    "server": "zabbix.example.com",
    "port": "10051",
    "collector_host": "HPE Aruba Central Collector",
    "sender_path": "zabbix_sender",
    "api_url": "https://zabbix.example.com/api_jsonrpc.php",
    "api_token": "zabbix-api-token",
    "unmapped_host_group": "HPE Aruba Central/Unmapped",
    "host_name_prefix": "",
    "managed_tag": {
      "tag": "hpe-aruba-central-ng",
      "value": ""
    },
    "host_tags": {
      "ap": "WiFi",
      "switch": "Switch",
      "gateway": "Gateway"
    },
    "templates": {
      "collector": "HPE Aruba Central NG - Collector",
      "site": "HPE Aruba Central NG - Site",
      "ap": "HPE Aruba Central NG - AP",
      "switch": "HPE Aruba Central NG - Switch",
      "gateway": "HPE Aruba Central NG - Gateway"
    }
  },
  "collector": {
    "interval_seconds": 300,
    "collect_client_counts": false,
    "lld_settle_seconds": 10,
    "version_check_enabled": true,
    "version_check_base_url": "https://api.github.com/repos/nk02/zabbix-aruba-central-ng/contents",
    "version_check_ref": "main"
  },
  "workspaces": [
    {
      "name": "WORKSPACE MSP",
      "mode": "msp",
      "workspace_id": "workspace-id",
      "client_id": "client-id",
      "client_secret": "client-secret",
      "central_base_url": "https://de2.api.central.arubanetworks.com",
      "tenant_allowlist": [],
      "tenant_mappings": [
        {
          "tenant_name": "Central tenant name",
          "host_prefix": "CUSTOMER"
        }
      ]
    },
    {
      "name": "WORKSPACE STANDALONE",
      "mode": "standalone",
      "workspace_id": "workspace-id",
      "client_id": "client-id",
      "client_secret": "client-secret",
      "central_base_url": "https://de2.api.central.arubanetworks.com",
      "mapping": {
        "host_prefix": "CUSTOMER"
      }
    }
  ]
}
```

Modes:

- `msp`: MSP workspace. The collector discovers tenants and exchanges tokens for each tenant.
- `standalone`: workspace is treated as a single tenant-like scope.

`tenant_allowlist` is optional. Leave it empty to monitor all MSP tenants, or list tenant IDs/names to limit collection.

`collect_client_counts` is disabled by default to reduce API calls and collection time. Enable it only if you need wireless client count per AP.

`lld_settle_seconds` controls how long `push-all` waits between sending discovery data and sending item values. This reduces first-run failed values while Zabbix creates low-level discovery item prototypes.

`version_check_enabled` adds collector and template version status to `collector.health`. The collector checks GitHub Contents API and raises update alerts only when a newer version exists.

Collector and template updates are reported through Zabbix health items. The collector does not self-update code or automatically import newer templates; run `import-zabbix-template --apply` and update the collector files during a controlled maintenance window.

`unmapped_host_group` is the staging Zabbix host group used when the collector creates new hosts. Keep it as a landing area, then move hosts to the final customer/site groups from Zabbix if needed. The collector does not change host group membership for existing managed hosts.

`host_name_prefix` is prepended to every Zabbix host managed by the collector. Leave it empty if each tenant/workspace mapping already provides the complete prefix.

`host_prefix` inside a tenant or workspace mapping controls the customer prefix used for all device and site hosts discovered in that tenant/workspace. Site-specific prefixes are intentionally not used; this keeps Zabbix API synchronization predictable and avoids accidental host group or naming drift.

Host identity is based on the calculated Zabbix host name. Changing a tenant or workspace display name in HPE GreenLake does not rename Zabbix hosts unless your configured `host_prefix` changes. Changing `host_prefix` makes the collector plan new host names; it does not automatically rename or delete the old hosts.

`managed_tag` is a safety guard for Zabbix API writes. The collector adds this tag to hosts it creates and refuses to update an existing host with the same name unless that host already has the same managed tag. The collector only creates or references `unmapped_host_group`; it does not create customer/site host groups.

`host_tags` controls Zabbix host tags created through the API. By default the collector writes one tag named `WiFi`, `Switch`, or `Gateway` with an empty value for device hosts. Collector and site hosts are not tagged by default. No workspace, tenant, or site host tags are added.

`workspaces.json` contains secrets and is ignored by Git.

Do not overwrite your real `workspaces.json` with the example after the first setup. The example file is versioned and may gain new recommended settings over time. The collector reports config drift through `collector.health`:

- `config_status.status`
- `config_status.current`
- `config_status.expected`
- `config_status.missing_recommended_paths`

If Zabbix raises `local workspaces.json should be reviewed`, compare only the missing paths with `workspaces.example.json` and add the new keys to your existing local file. Missing optional keys still use collector defaults, so collection keeps running.

## Commands

Validate config:

```powershell
python .\central_collector.py config-check
```

Validate authentication:

```powershell
python .\central_collector.py auth-check
```

Print workspace, tenant, and device summary:

```powershell
python .\central_collector.py summary
```

Print the planned collector, site, and device hosts without calling the Zabbix API:

```powershell
python .\central_collector.py plan-zabbix-hosts
```

Print the bundled template import plan:

```powershell
python .\central_collector.py import-zabbix-template
```

Import or update the bundled templates through the Zabbix API:

```powershell
python .\central_collector.py import-zabbix-template --apply
```

The template import creates or updates only the bundled HPE Aruba Central templates. It does not delete missing items, discovery rules, triggers, or value maps from existing templates.

Create or update Zabbix host groups and hosts through the Zabbix API:

```powershell
python .\central_collector.py sync-zabbix-hosts --apply
```

Host synchronization is guarded by the configured `managed_tag`. If an existing host has the same name but does not carry that tag, the collector refuses to update it. The collector creates or reuses only `unmapped_host_group` for new hosts; customer/site host groups and later host moves are intentionally left to Zabbix administrators.

Dry-run without sending to Zabbix:

```powershell
python .\central_collector.py push-all --dry-run
```

Send to Zabbix:

```powershell
python .\central_collector.py push-all
```

`push-all` synchronizes collector, site, and device hosts through the Zabbix API before sending values. New devices discovered in Central are therefore created automatically before `zabbix_sender` runs. Hosts for devices removed from Central are not deleted automatically; remove or disable those managed hosts manually after confirming they are no longer needed.

The collector also reports managed Zabbix hosts that are no longer discovered in Central through `central.collector.stale_managed_hosts_count`, `central.collector.stale_managed_host_names`, and `central.collector.stale_managed_hosts`. This raises a warning with the stale host names in operational data, so old hosts can be reviewed without deleting anything automatically.

Run continuously:

```powershell
python .\central_collector.py daemon --push-command push-all
```

Windows helper:

```powershell
.\start-collector-daemon.ps1
```

Windows scheduled task every 5 minutes:

```powershell
schtasks /Create /TN "HPE Central to Zabbix" /SC MINUTE /MO 5 /TR "powershell.exe -NoProfile -ExecutionPolicy Bypass -Command cd 'C:\zabbix\hpe-central-zabbix'; python .\central_collector.py push-all" /F
```

Windows daemon at logon:

```powershell
schtasks /Create /TN "HPE Central Collector Daemon" /SC ONLOGON /TR "powershell.exe -NoProfile -ExecutionPolicy Bypass -File 'C:\zabbix\hpe-central-zabbix\start-collector-daemon.ps1'" /F
```

Linux cron:

```cron
*/5 * * * * cd /opt/hpe-central-zabbix && /usr/bin/python3 central_collector.py push-all
```

## Sent Data

Collector health:

```text
central.collector.health
```

APs:

```text
central.ap.raw
```

AP radios:

```text
central.ap.radios.discovery
central.ap.radio.raw[<radio-number>]
```

Switches:

```text
central.switch.raw
central.switch.interfaces.discovery
central.switch.interface.raw[<port-index>]
central.switch.lags.discovery
central.switch.lag.raw[<lag-id>]
central.switch.stack_members.discovery
central.switch.stack_member.raw[<member-id>]
central.switch.hardware.discovery
central.switch.hardware.raw[<hardware-id>]
central.switch.vsx.raw
central.switch.hardware_trends.raw
```

AP and switch raw items are sent to the per-device host. Radio, interface, LAG, stack member, and hardware discovery is local to that device host. Switch optional data is sent only when Central returns data. Hardware categories include the component health data exposed by Central, such as fans, power supplies, CPU, memory, and temperature when available.

GreenLake subscriptions:

```text
central.licenses.discovery
central.license.raw[<scope>,<owner-id>,<subscription-id>]
```

The collector monitors only non-evaluation active subscription keys (`isEval=false` and status `STARTED`/`ACTIVE`). This avoids noise from expired evaluation subscriptions. The full subscription key is not sent to Zabbix; the subscription ID is used in item keys and only the key suffix is exposed for human recognition.

`collector.health` includes:

- `workspace_count`
- `tenants_count`
- `devices_total`
- `device_counts_by_type`
- `device_counts_by_workspace`
- `collector_version`
- `template_version`
- `version_status`
- `config_status`
- `sent_lines`
- `elapsed_seconds`

## Streaming APIs

New Central Streaming APIs can be useful, but they are a different flow from the REST polling used by this collector. Central exposes WebSocket endpoints such as `wss://<central-base-url>/network-services/v1alpha1/<endpoint-path>`, authenticated with the same OAuth bearer token used by REST APIs. Events are CloudEvents encoded with protobuf, so a streaming consumer must decode both the CloudEvent envelope and the event-specific payload.

Streaming is best implemented as a second daemon, not inside `push-all`:

- receive AP monitoring events in near real time;
- consume audit trail or configuration events;
- convert selected events to Zabbix trapper items or event-style problems;
- keep REST polling as the periodic inventory and metric baseline.

New Central Streaming APIs require devices with Central Advanced tier.

## Security Notes

The collector stores temporary access tokens in `.token_cache.json`. Protect the collector directory.

Never commit:

- `workspaces.json`
- `.token_cache.json`
- `zabbix_sender.exe`
